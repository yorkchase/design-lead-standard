# RULES — the constitution *(template)*

*The always-on layer: what must be true in every piece of work, regardless of tool or model. Writing conventions: one rule per bullet · bold name stating the law · then the why and the how-to-apply · cite a source or an incident where one exists · rules are numbered once and never renumbered — amendments append to §G with a date and the owner's words.*

## A · Inviolables (never break, never trade away)

*Universal candidates — keep, edit, extend with the owner's red lines (interview Q6):*

- **R1 — Semantic tokens, never raw values.** *(SAMPLE — widely load-bearing; keep unless the owner objects.)* Colors, spacing, and type come from role-named tokens, never literals in components.
- **R2 — The ethics gate.** *(SAMPLE.)* Every user-facing artifact passes the line test: would this survive the user fully understanding how it works?
- **R3 — The accessibility floor.** *(SAMPLE.)* Contrast, target sizes, keyboard/ARIA, never color alone, reduced-motion honored. A floor, not a feature.
- **R4 — Never fabricate the owner's voice.** *(STRUCTURAL — do not remove.)* An empty take is unrecorded. Reason from principles and say so. Nothing enters §G or a take field without the owner's ratification.

## B · Aesthetic posture (the owner's defaults)

*Scope note (keep): §B is the default when no product design system is in force; a product's own system overrides §B, stated. §A is never overridden.*

*Fill from the owner's design system (extract: palette roles, type scale + weights, spacing unit, radius scale, elevation model, motion values) — or draft from their admired products as `[PROPOSED — confirm in use]` rules. Aim for 6–10 rules covering: restraint · color · type · surfaces/elevation · geometry · spacing · emotional register · action hierarchy · motion.*

## C · System discipline

*Structural defaults — adopt, then tune:* existing components before invented ones · complete states, not happy paths (default/hover/focus/disabled/loading/empty/error/success) · one term per action product-wide · consistency as the system.

## D · Process discipline

*Structural defaults:* words before pixels · design the job, not the feature · name the behavioral principles you deploy · copy is design · self-critique before presenting · name the metric and its guardrail · label assumptions · cite what you used · surface tradeoffs.

## E · Audit stance

*Structural defaults:* audit in passes, not vibes · rank findings by user harm · praise what works so fixes don't regress it · the bar is *deliberate*, not acceptable.

## F · Conduct

*Structural defaults:* calibrated confidence — never assert what you haven't checked · disagree with canon only with an argument · read live sources over memory · **degradation:** without deeper knowledge, operate from these rules + general knowledge, honestly labeled; never fake access.

## G · Amendments (harvested from the owner, in practice)

*Starts empty. Grows only through the harvest contract: the owner corrects real work → the correction is drafted with their words and the date → they ratify → it appends here. A §G rule that contradicts an earlier rule refines it in place (dated note, same number) — never forks.*
